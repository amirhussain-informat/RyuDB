  What step 6 is (the design already agreed in memory)

  Persist each committed delta batch as a WAL record; commit_ts becomes the WAL LSN. On Catalog/Engine startup, replay the WAL to reconstruct the in-memory delta. Key shape:
  - WAL file per data dir (e.g. <data_dir>/ryudb.wal), append-only, fsync on commit.
  - _commit writes a record (table + serialized batch + the new commit_ts) before / as it appends to the in-memory delta (durability: recoverable up to the last fsynced LSN).
  - A RESTORE/rewind needs to truncate the WAL to the target LSN (the delta rewind(ts) already exists — mirror it on the WAL).
  - Recovery: on Engine init, scan the WAL and replay surviving records into DeltaStore (re-tag with their commit_ts), then set _commit_ts to the max replayed LSN.
  - Snapshots could optionally be a WAL checkpoint marker (or stay in-memory only — decide; memory currently says snapshots are in-memory, so a restart drops them, which is fine for step 6 scope).

  Critical context to keep in mind (from memory)

  - MVCC invariant is load-bearing: commit_ts monotonic + append-to-tail + _merge_delta concats batches_at ++ buffer in order → visible series byte-identical pre/post commit. The WAL replay MUST preserve batch order and commit_ts tagging or the over-invalidation safety breaks.
  - Single-session, in-memory txns today — a pending (uncommitted) txn is implicitly rolled back on process exit. The WAL only records committed batches (buffered/uncommitted frames are never written). Don't WAL the buffer.
  - _invalidate_table_caches is reused on every write/commit/rollback/restore — the WAL recovery path should rebuild caches lazily (they re-populate on first read), no need to persist them.
